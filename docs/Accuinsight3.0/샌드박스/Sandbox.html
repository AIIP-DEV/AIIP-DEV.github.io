샌드박스

Sandbox 선택​

샌드박스는 Rstudio 또는 Jupyter Lab과 같은 분석 툴이 설치된 가상 컴퓨팅 환경입니다.

프로젝트 생성 후 [샌드박스 탭]에서 [샌드박스 생성] 버튼을 통해 분석 환경 설정으로 이동하여 생성을 할 수 있습니다.

정보

[워크플로우]를 통해 Sandbox 생성 방법 및 관련 기능(데이터/모델 추출) 등과 같은 내용은 아래 매뉴얼 경로 참고

매뉴얼 경로: [워크플로우] -> [Pyspark 워크플로우 화면] -> [모델링] -> [sandbox]

Sandbox 분석 환경 설정​

샌드박스 생성에 필요한 설정들을 입력 후 샌드박스를 생성합니다.

  1. 분석 환경
    • 1-1) 베이스 모델 : Rstudio, Jupyter lab
    • 1-2) 샌드박스 애셋 선택
정보

샌드박스 애셋은 이전에 생성한 샌드박스(Rstudio, Jupyter)을 애셋으로 저장 후 동일한 환경으로 생성하는 기능
-> 샌드박스 애셋 저장 기능은 아래 [Sandbox 애셋 저장] 매뉴얼 참고

  1. 자원 설정

  • 2-1) 프로젝트 가용 자원

    • 프로젝트 가용 자원 범위 현황

  • 2-2) 클러스터 가용 자원

    • 클러스터 가용 자원 범위 현황

  • 2-3) 샌드박스가 사용할 자원을 Small, Medium, Large, 직접 입력 중 선택

    • 직접 입력을 선택하는 경우 Dropdown 메뉴를 통해서 CPU, Memory 설정
    • 해당 프로젝트에서 GPU 자원 사용이 가능하도록 설정 된 경우 GPU 설정을 추가
    • GPU 설정 추가 선택 후 Dropdown 메뉴를 통해서 GPU Core를 설정
  1. Rstudio / Jupyter / 샌드박스 애셋 비밀번호 설정
    • Rstudio / Jupyter Lab 접속 시 사용 할 비밀번호를 입력
    • 비밀 번호는 영문, 특수문자 조합 8자 이상만 입력

Sandbox 관리​

생성 된 Sandbox를 프로젝트 화면에서 확인 할 수 있습니다.

  1. 프로젝트에 속한 샌드박스 수를 표시
  2. 샌드박스 목록을 열거나 닫음
  3. 샌드박스 View 새로고침
  4. 샌드박스 생성
  5. 북마크 설정/해제
정보

북마크 설정 시 우선적으로 정렬되어 리스트 상위에 표시 됩니다.

  1. 샌드박스 의 상태 및 변경 확인
    • Pending : 샌드박스 환경 생성 대기 중
    • Waiting : 샌드박스 환경 생성 중
    • Running : 샌드박스 환경에 접속 가능한 정상 실행중인 상태
    • Stop : 샌드박스 환경이 중지 상태이며, 접속 불가능.
  2. 샌드박스의 사용자원 및 설정자원 확인
    • CPU: CPU 사용량/ CPU 설정량
    • GPU: GPU 사용량/ GPU 설정량
    • MEML: Memory 사용량/ Memory 설정량
  3. 샌드박스의 More Actions 메뉴
    • 중지 - 샌드박스 중지
      • 샌드박스 중지 시 해당 샌드박스 이미지 백업 후 시작 시 해당 백업 이미지 사용
    • 샌드박스 저장
      • 샌드박스 코드와 컨테이너 이미지를 저장하는 기능
    • 샌드박스 애셋 저장 - 샌드박스 애셋 저장 기능
      • 아래 [Sandbox 애셋 저장] 기능 참고
    • 삭제 - 샌드박스 삭제 수행

Sandbox 애셋 저장​

샌드박스 애셋 저장은 위 샌드박스 More Actions 중 '샌드박스 애셋 저장' 기능을 통해 애셋으로 저장할 수 있습니다.

  1. 애셋 이름 입력
  2. 소스 포함 여부 : 해당 샌드박스 내 소스 저장 여부
  3. 설명 : 샌드박스 애셋 설명 입력
  4. 공유범위 설정
    • 나만 보기 : 소스 저장소를 생성한 사용자만 공개
    • 프로젝트 내 공개 : 소스 저장소를 생성한 사용자가 포함된 프로젝트의 구성원에게 공개
    • 전체 공개 : 전체 구성원에게 모두 공개
정보

대시보드 화면 내 샌드박스 리스트에서도 동일하게 '샌드박스 애셋 저장' 사용 가능

Sandbox 상세 정보​

샌드박스 상세페이지는 해당 컨테이너의 상세 정보를 조회하고 분석툴에서 생성한 파일을 조회하고 저장소로 저장할 수 있습니다.

1. Sandbox 정보​

워크스페이스 생성시 설정된 정보와 현재 컨테이너 상태, 발생한 이벤트 정보를 조회합니다.
Endpoint URL 바로가기 (분석툴 새창 open)지원, Jupyter/R 환경 생성 시 설정한 비밀번호 숨김/확인 기능을 제공합니다.

  1. 분석 환경에 대한 정보 확인
    • 접속 정보 : Rstudio 또는 Jupyter lab 접속 패스워드
    • 컨테이너 이미지 : 샌드박스 컨테이너 이미지의 저장 경로 정보
  2. 샌드박스의 CPU, GPU, MEMORY 사용자원 및 설정자원 정보를 확인 및 수정
    • 자원 설정 : stop 상태일 때만 수정 가능
  3. 컨테이너의 상태 히스토리 확인
  4. 발생 이벤트 확인

2. Experiment​

Jupyter Lab 에서 실행된 (Accuinsight SDK 활용) experiment를 관리합니다.

  1. Experiment 별 결과 확인

  2. 동일한 지도학습 항목별로 2~10 이내 Experiment 에 대한 결과 비교 확인

3. 소스 폴더​

Sandbox 홈 디렉토리의 파일 목록 조회 및 파일 내용 확인을 하고 파일을 선택하여 소스 저장소로 저장할 수 있습니다.

  1. 선택한 파일의 내용을 확인 할 수 있습니다.
  2. 소스 저장소로 저장
    • 소스 저장소는 분석 작업중인 소스코드 형상관리를 위한 저장소이며 Sandbox가 삭제되더라도 소스 저장소에 저장된 소스코드는 유지됩니다.
    • 소스크기에 따라 저장하는 과정에서 다소 시간이 소요될 수 있습니다.
  3. 소스를 새로운 이름 또는 기존 저장된 소스에 새로운 버전을 발행하여 저장되며, 버전은 시스템에서 자동으로 발행됩니다.
  4. 저장하는 소스의 공유범위 설정
    • 나만 보기 : 소스 저장소를 생성한 사용자만 공개
    • 프로젝트 내 공개 : 소스 저장소를 생성한 사용자가 포함된 프로젝트의 구성원에게 공개
    • 전체 공개 : 전체 구성원에게 모두 공개

Experiment > AccuInsight SDK​

Experiment에 학습된 모델을 등록하기 위한 Python sdk인 Accuinsight 를 제공합니다.
Accuinsight 는 학습에 사용된 machine learning platform에 따라 3가지 api로 구분되며, 사용자 편의를 위한 utils 를 포함합니다.

  • ML API : Scikit-learn / Xgboost / lightGBM
  • tensorflow API : Tensorflow 2.0 (keras는 tf.keras 사용)
  • utils
정보

Tensorflow 2.0의 estimator로 작성된 모델과 Pytorch는 현재 지원하지 않습니다.

API 소개​

1. API 공통 기능​

(method) set_storage( access_key, secret_key, region, bucket_name, hdfs_uri, file_path, target, save_json, save_path )

accu.set_storage(file_path=/home/work/boston_data.csv, target='MEDV')
  • 모델링에 사용하고자 하는 데이터를 SDK에 등록하고, data drift 기능을 활용하기 위해서 필요한 작업입니다.
  • Accuinsight+의 Data Catalog와 HDFS, Amazon S3를 지원합니다. 또는 노트북에 존재하는 파일을 등록할 수도 있습니다.
  • 저장소 종류에 따라 필요한 인수(argument)가 다릅니다. 필수 인수는 file_path와 target 입니다.
  • 저장소 정보를 json file로 저장해 둘 수 있습니다. 이 json_file은 아래의 get_file() 메소드에서 활용됩니다.
  • json file을 저장할 경로를 지정할 수 있습니다. 지정하지 않을 경우 현재 경로에 저장합니다.
  • Arguments
    • access_key: AWS S3 저장소의 access key를 입력합니다 (AWS S3).
    • secret_key: AWS S3 저장소의 secret key를 입력합니다 (AWS S3).
    • region: AWS S3 저장소의 region을 입력합니다 (AWS S3).
    • bucket_name: AWS S3 저장소의 bucket name을 입력합니다 (AWS S3).
    • file_path: 저장소 내의 파일 경로를 입력합니다 (공통, 필수 argument).
    • target: target(y) 변수명을 입력합니다 (공통, 필수 argument).
    • save_json: 저장소 정보를 저장하고 싶을 경우, True로 지정합니다 (공통).
    • save_path: 저장소 정보를 저장할 경로를 지정합니다 (공통).

(method) get_file( storage_json_file_name )

accu.get_file(storage_json_file_name)
  • 모델링에 사용하고자 하는 데이터가 저장된 저장소에서 데이터를 다운로드 받습니다.
  • 저장소 정보를 json file로 넘겨줄 수 있습니다. json file이 입력되지 않으면 set_storage 메소드에서 등록한 데이터를 불러옵니다.
  • 다운로드 완료 후, 로그 메시지에 기록된 path를 사용하여 직접 데이터를 불러옵니다.
  • Arguments
    • storage_json_file_name: 다운로드 받을 데이터 정보를 담은 json file의 경로를 입력합니다 (선택 사항).
accu.get_file('/home/work/boston_data.csv')
정보

학습에 사용할 데이터가 노트북에 있는 데이터인 경우, 이 과정은 생략할 수 있습니다.

(method) set_features( feature_names = [feature_list])

accu.set_features(feature_names=['MEDV', ''])
  • 데이터를 불러온 후, 테이블 구조 변경으로 features 정보가 변경되었을 경우 사용합니다.
  • 노트북에서 데이터를 변경한 경우 변경한 테이블을 자동으로 다시 인식하므로, 이 메소드를 사용할 필요가 없습니다.
  • Arguments
    • feature_names: features list. python list 또는 pandas dataframe 형태로 넣어 주어야 합니다.

(method) set_slack( hook_url = 'slack_webhook_url')

accu.set_slack(hook_url='https://hooks.slack.com/services/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')
  • 모델 학습이 완료되거나, 모델 학습 도중 사용자가 설정한 조건을 만족하는 경우, Slack으로 메시지를 받아보실 수 있습니다.
  • set_slack() 메소드를 호출하여 slack 채널의 webhook url을 먼저 설정해주어야 합니다.
  • Arguments
    • hook_url: Slack 채널의 webhook url
  • slack webhook url은 slack workspace에 incoming webhook을 추가하면 가져올 수 있습니다.
  • slack webhook url로 메시지를 받을 채널이나 DM을 지정할 수 있습니다.
  • 여러 개의 채널에 알람을 보내고 싶을 경우, ',' 로 구분하여 넣어줍니다.

(method) send_message( message* = 'message', thresholds** = 0.7 )

# case 1 - 학습 완료 메시지를 푸시할 경우
accu.send_message(message='학습이 완료 되었습니다.')

# case 2 - metric이 thresholds 초과시 메시지를 푸시할 경우
accu.send_message(thresholds=0.7)
  • 위의 example과 같이, 경우에 따라 사용되는 Arguments 의 차이가 있습니다.
  • Arguments
    • message: 모델 완료시 받고자 하는 메시지
    • thresholds: 알람을 받고자 하는 metric의 임계치
정보

두 가지 Arguments 를 동시에 사용할 수 없습니다. ML experiment의 경우, thresholds argument를 사용할 수 없습니다. message argument를 사용해 주십시오.

(method) unset_slack()

accu.unset_slack()
  • slack alarm 설정을 해제합니다.
  • 이 메소드는 argument 없이 동작합니다.

(method) unset_message()

accu.unset_message()
  • alarm 메시지 설정을 해제합니다.
  • 이 메소드는 argument 없이 동작합니다.

2. ML API​

ML API example

(class) accuinsight()

from Accuinsight.Lifecycle.ML import accuinsight

accu = accuinsight()
  • accuinsight() class를 임의의 변수(accu)로 선언합니다.

(class) add_experiment( model_name, X_test, y_test, model_monitor, runtime )

y_pred = model.predict(X_test)
mse = sklearn.metric.mean_squared_error(y_test, y_pred)
mae = sklearn.metric.mean_absolute_error(y_test, y_pred) 

with accu.add_experiment(model_name, X_test, y_test, model_monitor=True, runtime=True) as exp:
  exp.log_params('alpha')
  exp.log_params('beta')
  exp.log_metrics('MSE', mse)
  exp.log_metrics('MAE', mae)
  exp.log_tag('model-test')
  • add_experiment 는 학습된 모델을 등록하는 데 사용되는 class로, with 구문에서 아래의 메소드를 수행합니다.
    • (method) log_params( 'hyperparameter name' )
      모델 학습에 사용된 모델의 hyperparameter를 기록합니다.
    • (method) log_metrics( 'metric name' , defined_metric )
      모델로 예측을 수행한 후, 계산된 metric(defined_metric)을 기록하고자 하는 이름('metric name')과 함께 입력해줍니다.
    • (method) log_tag( 'model description' )
      모델에 대한 간단한 설명을 기록합니다.
  • Arguments
    • model_name: 학습된 모델의 이름
    • X_test: 평가에 사용될 input dataset의 이름
    • y_test: 평가에 사용될 target dataset의 이름
    • model_monitor: Data drift 기능을 사용할 경우, model_monitor=True 옵션을 사용하여 피처 중요도를 저장합니다. (비정형 데이터 분석 시 model_monitor=False)
    • runtime: AIRuntime을 이용한 모델 배포를 사용할 경우, runtime=True 옵션을 사용합니다.
정보
Scikit-learn의 SVM으로 모델링을 진행할 경우, 다음을 유의하시기 바랍니다.
  • multiclass classification에서 SVC 를 사용하여 모델링 할 경우, probability = True 옵션 설정이 필요합니다.
  • multiclass classification에서 LinearSVC 를 사용하여 모델링 한 경우, predict_proba() 를 제공하지 않아 ROC/Precision-recall curve가 제공되지 않습니다.

3. tensorflow API​

tensorflow API example

정보
keras API는 tensorflow API 구조가 동일합니다.

(class) accuinsight()

from Accuinsight.Lifecycle.tensorflow import accuinsight

accu = accuinsight()
  • accuinsight() class를 임의의 변수(accu)로 선언합니다.

(class) autolog( tag = 'model description', best_weights, model_monitor, runtime )

accu.autolog(tag = 'tensorflow-test', best_weights = True | False, model_monitor = True | False, runtime = True | False)
  • autolog 는 keras의 fit() 메소드를 수행하기 이전에 선언한 후, fit() 을 통해 모델을 학습시키면, 학습 과정에서 자동으로 Experiment를 등록합니다.
    • autolog 를 선언하면, 모델 학습 종료 시점에 가장 좋은 metric을 갖는 epoch의 weight가 자동으로 설정됩니다. 단, 이 때 fit() 메소드에 입력된 epoch만큼 학습이 진행되므로 빠른 학습 종료를 위해서는 early_stopping 콜백을 추가해야 합니다.
    • autolog 는 가장 좋은 metric을 기록한 epoch에서의 모델 정보와 weight를 저장합니다. 따라서 이전에 저장된 모델을 불러와 공동 작업자들과 공유하거나, 재학습을 수행할 수 있습니다. 모델을 불러오는 메소드는 utils 항목을 참고하세요.
  • Arguments
    • tag (optional): 모델에 대한 간단한 설명
    • best_weight: 학습을 여러 번 진행할 경우, best_weight=True으로 설정 시 설정한 지표 중 가장 우수한 차수의 모델을 best_model로 저장, best_weight=True으로 설정 시 마지막으로 학습된 모델을 best_moderl로 저장합니다.
    • model_monitor: Data drift 기능을 사용할 경우, model_monitor=True 옵션을 사용하여 피처 중요도를 저장합니다. (비정형 데이터 분석 시 model_monitor=False)
    • runtime: AIRuntime을 이용한 모델 배포를 사용할 경우, runtime=True 옵션을 사용합니다.

model.compile() 시, metrics 리스트에서 처음으로 입력 된 평가지표를 활용하여 모델을 훈련합니다. 따라서 모델 훈련 시 기준이 되는 평가 지표를 먼저 입력해야 합니다. 예를 들어 아래와 같이 모델을 컴파일 한 경우, 모델 훈련 시 f1_score를 참조하여 가장 좋은 f1_score를 기록한 모델을 기록하고, 적합시킵니다. 

model.compile(metrics = [f1_score, accuracy])
